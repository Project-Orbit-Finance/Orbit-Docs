# ADR-001 — Decisões Arquiteturais Iniciais

## Status

Aceito

## Contexto

Após a análise técnica do projeto Orbit, foram identificados alguns pontos que precisavam de definição antes do início da implementação.

Este documento registra as decisões arquiteturais que passam a ser consideradas oficiais para o projeto.

## 1. Autenticação

A autenticação será responsabilidade do backend Laravel.

O frontend nunca acessará o banco diretamente.

O fluxo será:

React → Laravel → PostgreSQL

O Laravel será responsável por autenticação, autorização, emissão e renovação de tokens.

## 2. Arquitetura Frontend

Será utilizada arquitetura Feature-Based.

Cada feature conterá:

* components
* hooks
* services
* schemas
* types

Componentes nunca acessarão APIs diretamente.

Hooks serão responsáveis pelo estado e orquestração.

Services serão responsáveis pela comunicação com o backend.

Regras matemáticas puras permanecerão em `utils/` ou em módulos específicos de domínio.

## 3. Categorização

A primeira versão utilizará regras determinísticas.

Cada transação será analisada por palavras-chave.

Correções realizadas pelo usuário gerarão novas regras personalizadas para aquele usuário.

Nenhum algoritmo de Machine Learning será utilizado no MVP.

## 4. Modelo de Dados

As importações serão armazenadas separadamente das transações.

Cada transação poderá estar vinculada a uma importação.

Categorias serão entidades independentes referenciadas por chave estrangeira.

## 5. Auditoria

Todas as tabelas possuirão timestamps.

Eventos relevantes serão registrados para fins de rastreabilidade.

Não serão armazenados dados financeiros sensíveis em logs.

## 6. Escopo do MVP

O MVP conterá apenas:

* autenticação;
* dashboard;
* importação de CSV;
* gerenciamento de transações;
* categorização automática;
* gráficos financeiros;
* metas financeiras;
* simulador de investimentos;
* notificações básicas.

Funcionalidades como Open Finance, IA, OCR e sincronização bancária ficam explicitamente fora do escopo desta primeira versão.
